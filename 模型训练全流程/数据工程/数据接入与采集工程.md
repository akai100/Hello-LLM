## 企业 LLM 数据采集 ≠ 拉数据

你在这一阶段要解决的是 工程 + 治理问题。

**你必须同时满足 4 个条件**

| 条件  | 含义      |
| --- | ------- |
| 完整性 | 不漏、不截断  |
| 幂等性 | 重跑不重复   |
| 可追溯 | 知道从哪来   |
| 合规性 | 权限 & 脱敏 |

📌 如果一个数据源不满足这 4 条，**先不要急着用它做 LLM。**

## 企业常见数据接入类型

### 1️⃣ 结构化数据（DB / 表）

**典型来源**

+ 客服工单表

+ 用户反馈表

+ 业务日志表

**关键设计点**

+ 主键（唯一标识）

+ 时间字段（增量拉取）

+ 状态字段（是否有效）

📌 对 LLM 特别重要的一点：**不要只拉“最终结果”，要拉“过程字段”**

### 2️⃣ 半结构化数据（JSON / 日志）

**典型问题**

+ 字段不稳定

+ Schema 演进

+ 历史数据不一致

**企业级做法**

+ Schema 宽松

+ 原始字段全保留

+ 新字段向前兼容

### 3️⃣ 非结构化数据（文档 / 对话）

这是**LLM 最核心的数据来源。**

**常见类型**

+ PDF / Word / PPT

+ Wiki / Markdown

+ 聊天记录

📌 此时你的目标不是“解析得多漂亮”，而是：

**“信息是否完整保留下来？”**

## 企业级接入设计

### 1️⃣ Raw 数据永远不可逆

你必须做到：

```
原始数据（Raw）
 + 最少处理
 + 可回放
```

**允许的操作**

+ 格式统一

+ 字符编码修正

+ 基础脱敏（可选）

**禁止的操作**

+ 语义改写

+ 内容裁剪

+ 主观过滤

### 2️⃣ 增量优先，避免全量依赖

你要问数据源负责人三个问题：

1. 有没有时间字段？

2. 有没有唯一 ID？

3. 有没有删除 / 更新标志？

📌 企业里最怕的是：**“每天全量拉一遍，谁也不敢删”**

### 3️⃣ 幂等性

**问题场景**

+ 任务失败重跑

+ 重复写入

+ 训练数据悄悄变多

**解决思路**

+ 唯一键

+ 去重策略

+ 写前校验

### 4️⃣ 数据血缘

至少要能回答：

+ 这条数据来自哪个系统？

+ 哪一天采集？

+ 被谁用过？

📌 血缘 ≠ 复杂系统

一个 metadata 字段就能救命
