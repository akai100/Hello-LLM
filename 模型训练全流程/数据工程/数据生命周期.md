## 数据产生

**来源**

+ 用户对话

+ 内部文档

+ 业务系统日志

+ 人工标注

+ 专家知识

你要问的不是：数据干不干净？

而是：**这批数据，代表了哪一类真实世界分布？**

## 数据采集 & 接入

关键不是“能不能拉下来”，而是👇

| 维度  | 企业级要求 |
| --- | ----- |
| 完整性 | 不丢数据  |
| 幂等性 | 不重复   |
| 可追溯 | 知道从哪来 |
| 权限  | 合规    |

📌 一个企业级细节：**永远保留原始数据，不可逆处理永远在后面**

## 原始数据层

这是**不可碰**的层。

特点：

+ 不做语义修改

+ 不做业务判断

+ 只做轻量格式化（JSON / Parquet）

📌 为什么重要？

+ 数据审计

+ 问题回溯

+ 新策略重跑

## 清洗 & 规范化

你要做的是**分类，而不是裁决**。

常见操作：

+ 编码统一

+ 语言检测

+ 明显脏数据标记

+ 格式规范

🚫 不建议在这一层做的事：

+ 语义删除

+ 主观质量判断

## 语义加工 & 结构化

这里数据才开始“为模型服务”。

典型操作：

+ 文本拆分（Chunking）

+ 生成 instruction / QA

+ 自动标签

+ 元数据补充

📌 这一层要明确回答：**“这份数据，是为增强哪种模型能力？”**

## 数据版本化（企业分水岭）

企业级系统必须能回答：

+ 这个模型用了哪一版数据？

+ 如果效果下降，能否回滚？

+ A/B 测试差异来自哪里？

你至少要做到：

+ 数据快照

+ 版本号

+ Schema 固定

📌 没有数据版本，就没有工程化 LLM

## 使用层

同一份数据，不同做法

| 使用方式 | 数据要求    |
| ---- | ------- |
| RAG  | 可检索、可溯源 |
| 微调   | 高质量、低噪声 |
| 评测   | 稳定、小而精  |

**📌 不要用一份数据，硬撑所有用途**

## 效果反馈 & 数据回流

真正成熟的系统，一定有

```
模型失败
 → 样本收集
 → 失败归因
 → 数据增强
 → 再训练 / 再构建
```

📌 企业护城河 = **失败样本积累能力**
